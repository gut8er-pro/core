## CRITICAL BUGS (block production)

### BUG-006: Visits data never saves — 400 API error
- **Severity:** CRITICAL
- **URL:** `/reports/:id/details/accident-info`
- **Report Type:** All
- **Expected:** Visit data (street, postcode, location, date, expert) saves on blur
- **Actual:** API returns 400. Visit `type` field is required (`z.enum`) but `getValues('visits')` sends visits without a `type` when the radio hasn't been selected. The Zod validation rejects the entire PATCH, killing **all pending changes in the same batch** (including expert opinion, mediator, etc.)
- **Root cause:** `handleFieldBlur` calls `saveField('visits', getValues('visits'))` which includes visits without required `type` field. Zod rejects.
- **Cascade:** Expert Opinion fields (expertName, fileNumber, mediator) also lost if saved in same batch as visits.
- **Fix needed:** Either make visit `type` optional in Zod, or filter out incomplete visits before sending, or don't batch visits with other fields.

### BUG-008: Invoice line items not persisted
- **Severity:** CRITICAL
- **URL:** `/reports/:id/details/invoice`
- **Report Type:** All
- **Expected:** Line items (description, rate, amount) persist after reload
- **Actual:** Line items completely lost after reload. "Add Row" creates items client-side but `handleFieldBlur` doesn't handle `lineItems` array fields — they're never sent to the API.
- **Fix needed:** Add line item save logic (like visits — save entire array on any field blur, or use separate API calls per item).
